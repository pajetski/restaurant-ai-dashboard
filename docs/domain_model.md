# DUDE Domain Model

## Purpose
This document defines the core business entities DUDE will use.

The goal is to normalize messy real-world business data into a clean, durable model that supports alerts, forecasting, recommendations, and execution.

DUDE starts with restaurant operations, so the initial domain model is restaurant-first.

## Core entity groups

### 1. Organization and location

#### Location
Represents a business location or operating unit.

Suggested fields:
- id
- name
- timezone
- status
- operating_hours
- created_at
- updated_at

### 2. Vendors and purchasing

#### Vendor
Represents a supplier or purchasing source.

Suggested fields:
- id
- name
- contact_name
- contact_email
- contact_phone
- lead_time_days
- active_status
- created_at
- updated_at

#### Invoice
Represents the header-level record for a supplier invoice.

Suggested fields:
- id
- vendor_id
- location_id
- invoice_number
- invoice_date
- subtotal
- tax
- total
- source_type
- raw_source_reference
- created_at

#### InvoiceLineItem
Represents a parsed line item from an invoice.

Suggested fields:
- id
- invoice_id
- raw_item_name
- normalized_inventory_item_id
- quantity
- unit
- unit_cost
- line_total
- created_at

#### ReceivingEvent
Represents inventory actually received into a location.

Suggested fields:
- id
- location_id
- invoice_id
- inventory_item_id
- quantity_received
- received_at
- created_at

### 3. Inventory and menu

#### InventoryItem
Represents a stocked ingredient, product, or supply item.

Suggested fields:
- id
- name
- sku
- vendor_id
- unit
- par_level
- reorder_threshold
- current_cost
- active_status
- created_at
- updated_at

#### MenuItem
Represents a customer-facing item sold through the business.

Suggested fields:
- id
- name
- category
- active_status
- created_at
- updated_at

#### RecipeItemLink
Maps a menu item to the inventory items it consumes.

Suggested fields:
- id
- menu_item_id
- inventory_item_id
- quantity_per_sale
- unit
- created_at
- updated_at

#### MenuChangeEvent
Represents a material change to a menu item or menu configuration.

Suggested fields:
- id
- menu_item_id
- location_id
- change_type
- old_value
- new_value
- effective_at
- created_at

### 4. Sales and operations

#### SalesEvent
Represents a point-of-sale event or aggregated sales record.

Suggested fields:
- id
- location_id
- menu_item_id
- quantity_sold
- sold_at
- source_system
- created_at

#### HoursChangeEvent
Represents a change to a location’s operating hours.

Suggested fields:
- id
- location_id
- old_hours
- new_hours
- effective_date
- created_at

#### ScheduleEvent
Represents a staffing or shift-related record.

Suggested fields:
- id
- location_id
- employee_identifier
- role
- shift_start
- shift_end
- scheduled_hours
- created_at

### 5. Alerts, forecasts, and recommendations

#### Alert
Represents an issue, anomaly, or exception that requires attention.

Suggested fields:
- id
- code
- severity
- message
- location_id
- related_entity_type
- related_entity_id
- created_at
- resolved_at

#### Recommendation
Represents a suggested action generated by DUDE.

Suggested fields:
- id
- type
- priority
- rationale
- location_id
- related_entity_type
- related_entity_id
- created_at
- status

#### Forecast
Represents a predicted future condition or expected metric.

Suggested fields:
- id
- forecast_type
- location_id
- related_entity_type
- related_entity_id
- horizon
- predicted_value
- confidence_score
- created_at

## Modeling principles
- Prefer canonical entities over raw source-specific data
- Preserve references back to raw source records when possible
- Keep source adapters separate from core entities
- Use event objects for meaningful business changes
- Support historical analysis and traceability
- Keep the model extensible for future industries

## Near-term implementation priority
The first entities to implement in code should likely be:
- Location
- Vendor
- InventoryItem
- Invoice
- InvoiceLineItem
- ReceivingEvent
- Alert
- Recommendation

These support the initial product wedge of intake, normalization, alerting, and operational direction.
